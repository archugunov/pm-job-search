# How to Practise PM Cases — Methodology

The five-drill progression from recognition → generation → performance. Do them in
order; don't skip ahead.

These drills are built around three frameworks (all documented in the sibling
`case-types-reference.md`):

- **FPSSL** — Frame → Pick → Probe → Solve → Land (product sense / "improve X" cases)
- **Diagnose-Validate-Fix-Land** — analytics / metric-movement cases
- **Clarify → NSM → Decompose → Validate** — metric-tree cases

This guide describes the *practice methodology* around them — how to drill, in what
order, and how to lock what you learn.

## The five drills (in order)

1. **MC rapid-recognition** — recognise senior vs junior answers, build vocabulary.
   *(Built: run `/pm-job-search:case-practice`.)*
2. **Free-form generation** — same calls, no options; you write, you get scored. *(Roadmap.)*
3. **Recorded self-drill** — full case solo, voice-memo, review timing/delivery. *(Roadmap.)*
4. **Live mock** — interviewer pushes back, injects data mid-case.
   *(Built: run the `interviewer-simulator` agent once you clear the MC gate.)*
5. **Worked-example walkthrough** — see what "senior" looks like end-to-end. *(Roadmap.)*

Each drill builds on the previous. By drill 4 you have the vocabulary (drill 1), can
generate from scratch (drill 2), have timed yourself at length (drill 3), and can now
practise handling real-time pressure.

## Why this order (don't reverse it)

The sequence is deliberate. Recognition before generation, generation before
performance:

1. **MC drill first** — locks vocabulary and recognition before generation is required.
   You can't fix a gap you don't have the language to name.
2. **Free-form second** — generates from scratch, surfaces the gaps the MC drill
   couldn't show.
3. **Self-drill third** — builds muscle memory at full case length, exposes
   timing and delivery issues.
4. **Live mock fourth** — adds pressure, interruption, and mid-case recovery skill.
5. **Worked example fifth** — shows what "senior" looks like, *after* you've earned the
   right to see it (so you measure your own attempts against it, rather than copying it
   cold).

**Common mistake: running the live mock first.** Without the vocabulary (drill 1) and
the generation muscle (drill 2), you wobble, the interviewer has to break character to
coach you mid-flow, and the simulation collapses. The result is the worst of both
worlds — you learn nothing, feel rattled, and lose confidence going into the real
round. Clear the MC gate before you book a mock.

## The MC drill (drill 1) in detail

**Purpose:** lock the *recognition* of senior vs junior answers. Build vocabulary.
Calibrate your intuition before generation is required.

**Format:**

- Questions are mixed across three foundational concepts: **NSM** (north star metric),
  **Segments**, and **Needs**.
- Delivered in **batches of 4** questions.
- Each question: 4 options, single-select.
- One option is the strongest; **three distractors each demonstrate a specific failure
  mode** (vanity metric, wrong segmentation lens, solution-disguised-as-need, etc.).
- After each batch: the right pick + the specific failure mode each distractor shows +
  the senior pattern being reinforced.

**Example question shape:**

> *Pick the strongest north star metric for a B2C freemium, gamified language-learning
> app.*
> - A) Daily active users → *vanity, gameable*
> - B) Daily streak completion rate → *gameable, doesn't measure learning*
> - C) Weekly active learners with ≥3 lesson completions → ✓ *behaviour + threshold + cadence*
> - D) Total lessons completed per week → *aggregate volume hides per-user mix*

**The Needs questions are designed harder on purpose.** A naïve Needs question pairs 1
JTBD-formatted option with 3 obvious distractors (solution-disguised, generic,
feature-gap) — which makes the JTBD option always the discriminably-longest one, so
you're recognising *format*, not *job quality*. The better design uses **4
JTBD-formatted options**, forcing you to discriminate on stakes-in-the-so-clause,
non-obviousness, and segment-specificity. Expect those.

**Feedback structure (Pattern 1)** — for each question, ~150 words:

- ✓ Why the right answer is right (one line).
- ✗ For each distractor: the specific failure mode it demonstrates (one line each).
- The senior pattern the question is teaching (one line at the batch level).

The distractors teach more than the right answer — read every one.

**The readiness gate:** don't book the live mock (drill 4) until you're scoring
**≥80%** on the MC drill. The progression is there to protect you; skipping the gate is
how the mock collapses.

## Cross-drill feedback patterns

These five patterns recur across the drills. Knowing what good feedback looks like
helps you self-assess on the roadmap drills you run solo.

### Pattern 1 — Right answer + every distractor failure mode (drill 1)

Don't settle for "C is right." The useful feedback is *why C is right + what specific
failure mode each wrong answer shows*. The distractors are the lesson.

### Pattern 2 — Score + named gap + model answer (drill 2)

Score the answer, name the specific gap with the framework principle you missed, then
read the model answer for comparison. Not a thumbs-up / thumbs-down — the named gap is
what you carry forward.

### Pattern 3 — Real-time intervention only for commitment (drill 4)

In the live mock, the interviewer interrupts **only** when you fail to commit ("which
segment specifically?"). Otherwise they stay in character. Everything else waits for
the debrief — that's what keeps the simulation real.

### Pattern 4 — Senior vs junior comparison (drill 5)

For each move in a worked example, see what the senior version looks like *and* what
junior candidates do instead, with the senior signal named explicitly. The contrast is
what makes the move stick.

### Pattern 5 — Anchor extraction

After every drill, extract 4-6 **anchors** to lock — short verbatim phrases you
memorise. Everything else is improvised. (See "Anchors" below.)

## Spaced practice — the 4-day plan

Map this schedule onto each case type in the run-up to a real interview. **You do all of
this on your own** — the plugin is text-only and does **not** record or play audio. The
voice-memo and listen-back steps below are things you do with your own phone; Claude
never hears or produces audio.

| Day | Activity | Time |
|---|---|---|
| **Day 1** (4 days out) | Read the framework + tactical sentences once. Close the doc. Voice-memo the openers (clarification + 3-clause picking) on your phone. Re-open and check gaps. | ~15 min |
| **Day 2** (3 days out) | Read the worked example once. Narrate one stage from memory. Re-open and check. Voice-memo it. Listen back for filler and reading-vs-speaking. | ~20-25 min |
| **Day 3** (2 days out) | Voice-memo the full walkthrough from memory. Aim for target time. Don't re-read between attempts. Re-open only after your second take. | ~30 min |
| **Day of** (15 min only) | Read the tactical sentences once. Don't practise. Be fresh, not over-rehearsed. | ~15 min |

The principles underneath: **retrieval beats re-reading**, **speak don't read**, and
**anchors not scripts**. Every drill should end with a voice-memo step, a listen-back
step, and an updated anchor list.

## Anchors

After every drill, lock **4-6 anchors** — short phrases (5-15 words each) you memorise
verbatim. Everything else in the case is improvised live. Anchors are the load-bearing
sentences you can fall back on under pressure; scripting the whole answer makes you
sound rehearsed and brittle, but anchors give you fixed points to navigate between.

Example anchors:

- *"Trial vs retention is two different problems."*
- *"Designed for context-lacking, adopted by context-rich."*
- *"Smooth decay rules out trust and quality."*

Keep a running anchor list across drills. The discipline is: if you can't say it from
memory, it's not an anchor yet — re-drill it.
